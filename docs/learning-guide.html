01. The Big Picture — What & Why

Insurance companies have run on DB2 mainframes since the 1980s. These systems are reliable but expensive ($10,000+/month in MIPS costs), hard to hire for (COBOL developers are retiring), and can't integrate with modern cloud services.

This engine moves data from DB2 to FAST (Financial Administration Service Technology), a modern cloud platform, with these guarantees:

  • Zero downtime — production never goes dark during migration
  • 99.99% accuracy — every record verified field-by-field after migration
  • Full lineage — every field traced from source to target with transformation history
  • Instant rollback — if anything goes wrong, restore previous state in seconds
🏠 Real-World Analogy: Moving to a New House

Imagine moving your entire life from an old house (DB2) to a new one (FAST). You can't just throw everything in a truck — you need to pack carefully (extract), label every box (transform), check nothing broke during the move (reconcile), and keep living in the old house until the new one is fully set up (zero downtime). If the new house has a gas leak, you move right back (rollback).

03. DB2 Mainframe Extraction

What is DB2?

IBM DB2 is a relational database that runs on mainframes (z/OS). Insurance companies store policy, claims, and premium data here — often 50+ years of history. We connect via ibm_db Python driver and pull data using SQL with OFFSET/FETCH pagination.

DIAGRAM 4: DB2 Extraction Flow
  DB2 Mainframe (z/OS)
  ┌─────────────────────────────────────────────────┐
  │                                                 │
  │  SELECT * FROM POLICY_MASTER                    │
  │  ORDER BY POLICY_ID                             │
  │  OFFSET 0     FETCH FIRST 10000 ROWS ONLY  ──────► Batch 0 (10K records)
  │  OFFSET 10000 FETCH FIRST 10000 ROWS ONLY  ──────► Batch 1 (10K records)
  │  OFFSET 20000 FETCH FIRST 10000 ROWS ONLY  ──────► Batch 2 (10K records)
  │  ...                                            │
  │                                                 │
  └─────────────────────────────────────────────────┘
              │
              ▼
  ┌─────────────────────────────────────────────────┐
  │  For each batch:                                │
  │  1. Execute SQL query                           │
  │  2. Fetch rows as Python dicts                  │
  │  3. Compute XXHash checksum                     │
  │  4. Return ExtractionResult(records, checksum)  │
  └─────────────────────────────────────────────────┘

Why XXHash instead of SHA-256?

DIAGRAM 5: Checksum Speed Comparison
  Hashing Speed (higher = better):

  XXHash (xxh64):  ████████████████████████████████████████  10 GB/s
  CRC32:           ████████████████████████                   6 GB/s
  SHA-256:         ██████                                   500 MB/s
  MD5:             ████████████                               1 GB/s

  We use XXHash because:
  • 20x faster than SHA-256
  • We need INTEGRITY VERIFICATION, not cryptographic security
  • Millions of records → speed matters
  • If the checksum at source ≠ checksum at target, data was corrupted in transit

Parallel Extraction

A single DB2 connection is slow for large tables (millions of rows). We use Python's ProcessPoolExecutor to run multiple extraction workers in parallel, each pulling a different offset range.

DIAGRAM 6: Parallel Extraction with ProcessPoolExecutor
  Main Process
  ┌──────────────────────────────────────────────────────┐
  │  ParallelExtractor.extract_table()                   │
  │                                                      │
  │  Total: 100,000 records / batch_size: 10,000         │
  │  = 10 batches to extract                             │
  │                                                      │
  │  ProcessPoolExecutor(max_workers=4)                  │
  │  ┌──────────┬──────────┬──────────┬──────────┐       │
  │  │ Worker 1 │ Worker 2 │ Worker 3 │ Worker 4 │       │
  │  │Batch 0,4,│Batch 1,5,│Batch 2,6,│Batch 3,7 │      │
  │  │  8       │  9       │          │          │       │
  │  └────┬─────┴────┬─────┴────┬─────┴────┬─────┘      │
  │       ▼          ▼          ▼          ▼              │
  │  ┌─────────────────────────────────────────────┐     │
  │  │  Collect results + combine checksums         │     │
  │  │  Return: (all_records, combined_checksum)    │     │
  │  └─────────────────────────────────────────────┘     │
  └──────────────────────────────────────────────────────┘

  Why ProcessPoolExecutor (not ThreadPoolExecutor)?
  → DB2 extraction is CPU-bound (parsing EBCDIC, packed decimals)
  → Threads share GIL → no real parallelism for CPU work
  → Processes bypass GIL → true parallel execution
📦 Real-World Analogy: Warehouse Packing

Imagine a warehouse with 100,000 boxes to pack. One person (single thread) would take forever. Instead, you hire 4 workers (processes), each packing their assigned section simultaneously. Each worker stamps their boxes with a serial number (checksum) so you can verify nothing went missing.

# Key extraction code (simplified)
class DB2Extractor:
    def extract_batch(self, table_name, batch_size, offset):
        sql = f"""
            SELECT * FROM {table_name}
            ORDER BY POLICY_ID
            OFFSET {offset} ROWS
            FETCH FIRST {batch_size} ROWS ONLY
        """
        rows = self._execute_query(sql)
        checksum = xxhash.xxh64(str(rows).encode()).hexdigest()
        return ExtractionResult(records=rows, checksum=checksum)

Why Parquet?

DIAGRAM 8: File Format Comparison
  Format     Storage    Read Speed    Schema?    Spark?    Compression
  ───────────────────────────────────────────────────────────────────
  CSV        █████████  ████          No ❌      Slow      Poor
  JSON       ████████   ███           Partial    Slow      Poor
  Parquet    ██         █████████     Yes ✅     Fast ⚡   Excellent
  Avro       ███        ██████        Yes ✅     Good      Good

  Parquet wins because:
  ┌─────────────────────────────────────────────────────────────┐
  │  1. COLUMNAR STORAGE: Read only the columns you need       │
  │     CSV reads ALL columns even if you need just "name"     │
  │     Parquet skips irrelevant columns → 10-100x faster      │
  │                                                            │
  │  2. SCHEMA PRESERVATION: Types survive the round-trip      │
  │     CSV: "1250.50" (string? float? who knows)              │
  │     Parquet: 1250.50 (DOUBLE, guaranteed)                  │
  │                                                            │
  │  3. COMPRESSION: Snappy = fast + small                     │
  │     100MB CSV → ~10MB Parquet with Snappy                  │
  │     10x smaller files = 10x cheaper S3 storage             │
  │                                                            │
  │  4. SPARK-NATIVE: AWS Glue (Spark) reads Parquet natively  │
  │     No parsing overhead, direct memory mapping             │
  └─────────────────────────────────────────────────────────────┘
📚 Real-World Analogy: Library Organization

CSV is like a bookshelf where every book is face-out — you see everything but waste space. Parquet is like a card catalog system — books are stored spine-out (compressed), organized by category (columns), with an index card (schema) telling you exactly what's inside without opening the book.

After transformation and validation, records are pushed to the FAST cloud platform via its REST API. Records are sent in batches of 500 with automatic retry and exponential backoff.

  │  ┌─────────┐    fail    ┌─────────┐   fail   ┌─────────┐ │

DIAGRAM 16: Exponential Backoff Timing
  Attempt   Wait Before Retry   Total Elapsed
  ─────────────────────────────────────────────
  1         —                   0s
  2         2^1 = 2 seconds     2s
  3         2^2 = 4 seconds     6s
  (max)     RAISE EXCEPTION     6s

  Why exponential backoff?
  • Gives the server time to recover
  • Prevents "thundering herd" (all clients retrying at once)
  • 2^n growth means we wait longer each time
  • After max_retries, we give up and let the pipeline handle it

08. Blue/Green Deployment Strategy

This is the key to zero-downtime migration. FAST has two deployment "slots": Blue (production) and Green (staging). We load new data into Green, validate it, then atomically swap Green → Production.

  │   (PRODUCTION) 🟢       │    │   (EMPTY/STAGING)        │
  │                         │    │                          │

  │  Current live data      │    │                          │

  │      BLUE SLOT          │    │     GREEN SLOT           │

  │                         │    │                          │

🎭 Real-World Analogy: Theater Understudies

Blue/Green is like a theater with two stages. The audience watches the main stage (Blue/production). Meanwhile, the new act rehearses on the back stage (Green/staging). Once the director (reconciliation) approves, they swap stages instantly — the audience barely notices. If the new act forgets their lines, the old act is still ready on the back stage.

09. Record-by-Record Reconciliation

After loading, we compare every single record field-by-field between source (DB2 extract) and target (FAST loaded data). This is the safety net that ensures no data was lost or corrupted.

DIAGRAM 21: Acceptance Criteria
  ReconciliationReport:
  ┌─────────────────────────────────────────────────────────┐
  │  total_source:     100,000                              │
  │  total_target:     100,000                              │
  │  matched:          99,992                               │
  │  mismatched:       8                                    │
  │  missing_in_target: 0       ← CRITICAL: must be ZERO   │
  │  missing_in_source: 0                                   │
  │                                                         │
  │  match_rate: 99.992%                                    │
  │                                                         │
  │  is_acceptable = (match_rate >= 99.99%)                 │
  │                  AND (missing_in_target == 0)           │
  │                                                         │
  │  ✅ 99.992% >= 99.99% AND 0 missing = ACCEPTABLE       │
  │                                                         │
  │  ❌ If match_rate < 99.99% OR any missing → ROLLBACK   │
  └─────────────────────────────────────────────────────────┘

  Why 99.99% and not 100%?
  • Floating point precision differences between DB2 packed decimal
    and FAST's double representation cause tiny rounding diffs
  • 0.01% tolerance = 10 mismatches per 100,000 records
  • These are logged and reviewed, but don't block promotion
  • But ZERO tolerance for MISSING records (data loss is unacceptable)
📋 Real-World Analogy: Moving Company Inventory

After movers deliver your boxes, you check the inventory sheet item by item. A scratch on a lamp (value mismatch)? Annoying but acceptable. A missing wedding album (missing_in_target)? Unacceptable — stop everything, find it. That's our 99.99% match with zero missing tolerance.

DIAGRAM 24: Lineage Buffer & Flush Pattern
  Track entries accumulate in memory buffer:

  track() → buffer: [entry1]
  track() → buffer: [entry1, entry2]
  track() → buffer: [entry1, entry2, entry3]
  ...
  track() → buffer reaches batch_size (1000)
         → AUTO-FLUSH to PostgreSQL
         → buffer: [] (empty)
  track() → buffer: [entry1001]
  ...

  Why buffer?
  • Individual INSERTs to PostgreSQL = slow (1ms each × 1M = 1000 seconds)
  • Bulk INSERT 1000 at once = fast (50ms × 1000 batches = 50 seconds)
  • 20x faster with buffering
  • flush() called automatically at batch_size OR manually at end of job

16. Interview Questions & Answers

Q1: Why did you choose a 7-stage pipeline instead of a simpler 3-stage ETL?
Each stage serves a distinct purpose that can't be safely merged. Extraction is decoupled from transformation via S3 (allows re-running transforms without re-extracting). Validation is separate from transformation because business rules change independently from field mappings. Reconciliation is separate from loading because we need to compare source vs target, not just check loading errors. The pipeline is designed for auditability — each stage produces artifacts that can be independently verified.
Q2: How does the blue/green deployment achieve zero downtime?
Production always serves from the active slot (initially Blue). New data loads into the inactive slot (Green). Users are never affected during loading because they're reading from Blue. Promotion is an atomic pointer swap — one moment users read Blue, the next they read Green. If issues are found post-promotion, we swap back to Blue (which still has the old data). The entire operation is like flipping a switch, not a gradual migration.
Q3: Why 99.99% match rate instead of 100%?
DB2 uses packed decimal (BCD) for currency amounts, while FAST uses IEEE 754 floating point. The conversion can introduce tiny rounding differences at the last decimal place (e.g., 1250.5001 vs 1250.5002). We set abs_tol=0.001 for numeric comparisons. However, we have ZERO tolerance for missing records — any record present in source but absent in target is a critical failure that blocks promotion.
Q4: What's the difference between forward and backward lineage?
Forward lineage answers "if I change source column X, what target fields are affected?" — useful for impact analysis before schema changes. Backward lineage answers "where did target field Y come from?" — useful for auditors investigating a specific value. Both are built from the same lineage entries, just queried in opposite directions.
Q5: Why XXHash instead of SHA-256 for integrity checksums?
We need integrity verification, not cryptographic security. XXHash runs at 10GB/s vs SHA-256's 500MB/s — 20x faster. When processing millions of records per batch, this difference matters. We're checking "did the data arrive intact?" not "is this data tamper-proof?" If we needed tamper resistance (e.g., regulatory proof), we'd use SHA-256.
Q6: Why ProcessPoolExecutor instead of ThreadPoolExecutor for parallel extraction?
DB2 extraction involves CPU-bound work: parsing EBCDIC encoding, converting packed decimals, computing checksums. Python's GIL prevents true parallel CPU execution with threads. ProcessPoolExecutor spawns separate OS processes, each with their own GIL, enabling true parallel computation. The trade-off is higher memory usage (each process has its own memory space) and more expensive inter-process communication.
Q7: How does the rollback manager handle post-promotion failures?
Two scenarios: (1) Pre-promotion failure: simply discard the Green slot — zero impact since production is still on Blue. (2) Post-promotion failure (rare): emergency_swap() promotes Blue back to production. The old data is still in Blue because blue/green keeps both versions. This takes seconds. The key insight is that Blue slot isn't deleted after promotion — it becomes the new staging area, preserving the fallback.
Q8: Why use RAG for the chatbot instead of fine-tuning GPT-4?
Fine-tuning requires retraining the model every time lineage data changes (every migration job). RAG retrieves fresh data from ChromaDB at query time — always up to date. Fine-tuning is also expensive and slow (hours), while RAG indexing takes seconds. RAG also provides verifiable sources — we can show the user exactly which lineage entries the answer is based on, preventing hallucination.
Q9: How does the system handle schema changes in DB2 or FAST?
The entity_mappings table decouples source and target schemas. When a DB2 column is renamed, we update the mapping (source_column). When FAST adds a new field, we add a new mapping row. The transformation_rule column defines how to convert between formats. This means schema changes are configuration changes, not code changes — no redeployment needed for most mapping updates.
Q10: What happens if the Glue job fails mid-transformation?
The Glue transformer polls job status every 30 seconds. On FAILED status, it raises an exception. The orchestrator catches this, marks the job as "failed" in the metadata store, and triggers rollback (which cleans up any S3 staging data). Since we haven't loaded to FAST yet, there's nothing to rollback on the FAST side. The raw extracted data is still in S3, so we can retry transformation without re-extracting.
Q11: How do you ensure data lineage is captured for every field?
The FieldTransformer's transform_dataframe() method returns both the transformed DataFrame and a list of lineage entries. For every column that has a registered transform, it records the source column, source value, target column, target value, and transformation name. The orchestrator feeds these entries to the LineageEngine, which buffers them and bulk-inserts to PostgreSQL.
Q12: Why Parquet over CSV for S3 staging?
Three reasons: (1) Schema preservation — CSV loses type information, Parquet preserves it. A column that was DECIMAL(10,2) in DB2 stays numeric in Parquet. (2) Size — Parquet with Snappy compression is ~10x smaller than CSV. With millions of records, this saves significant S3 costs and transfer time. (3) Spark performance — AWS Glue (Spark) reads Parquet natively via memory mapping, while CSV requires parsing every row.
Q13: How would you scale this for billions of records?
Three scaling strategies: (1) Horizontal extraction — increase ProcessPoolExecutor workers and run against DB2 read replicas. (2) Glue scaling — increase DPU allocation (from G.1X to G.2X) and worker count (10→100). Spark auto-partitions the data. (3) Partitioned reconciliation — reconcile by partition (date range, region) instead of full-table compare. The architecture already supports this — each component is stateless and can process any partition independently.
Q14: What monitoring and observability do you have?
Structlog provides structured JSON logging with context (job_id, batch_number, duration_ms) at every stage. The audit_events table records every significant action. The metadata store tracks per-batch timing and error counts. In production, we'd add Prometheus metrics (extraction_duration_seconds, reconciliation_match_rate) and Grafana dashboards, plus PagerDuty alerts for failed jobs or low match rates.
Q15: Why Pydantic Settings instead of environment variables directly?
Pydantic Settings provides validation, type coercion, and documentation in one place. os.getenv("BATCH_SIZE") returns a string and can be None. Pydantic gives us settings.batch_size as a validated integer with a default, raising clear errors if required variables are missing. The @lru_cache on get_settings() ensures we parse environment variables once, not per-request.
Q16: How does the reconciler handle duplicate keys in the source?
When set_index("_recon_key") encounters a duplicate key, loc[key] returns a DataFrame instead of a Series. The code checks isinstance(source_row, pd.DataFrame) and takes iloc[0] (first occurrence). This is a conscious design choice — in insurance data, policy_id should be unique. Duplicates indicate a data quality issue that should be flagged in validation, but the reconciler handles it gracefully rather than crashing.
Q17: What's the purpose of the JSONB config column on migration_jobs?
Different migration jobs have different configuration: batch sizes, tolerance thresholds, column mappings, Glue DPU settings. Rather than adding a column for every possible config option (which would require schema migrations), we store the config as JSONB. This makes the migration_jobs table flexible — a policy migration and a claims migration can have completely different configs without schema changes.
Q18: How does exponential backoff prevent thundering herd?
If 100 batch uploads fail simultaneously (e.g., FAST rate limit), without backoff they'd all retry at once — making the problem worse. With exponential backoff (2s, 4s, 8s...), each retry waits longer. Adding jitter (random offset) means the 100 retries spread out over time instead of hitting simultaneously. Our implementation uses time.sleep(2 ** attempt) — simple but effective.
Q19: Why do you buffer lineage entries instead of inserting immediately?
A single migration job can generate millions of lineage entries (one per field per record). Individual PostgreSQL INSERTs take ~1ms each. 1M entries × 1ms = 1000 seconds (17 minutes) of just lineage writes. Buffering 1000 entries and doing bulk INSERT takes ~50ms per batch. 1000 batches × 50ms = 50 seconds. That's a 20x performance improvement from a simple batching pattern.
Q20: What security considerations are built in?
SSN masking ensures PII doesn't propagate to the target system. API authentication uses Bearer tokens for FAST API calls. The .env.example documents required secrets without exposing them. Docker health checks verify the service is running. In production, we'd add: encrypted S3 buckets (SSE-S3), VPC endpoints for S3/Glue (no public internet), IAM roles with least privilege, and audit logging of all API access.
Q21: How would you add a new entity type (e.g., agents)?
Three steps: (1) Add field mappings to entity_mappings table (source columns → target columns with transform rules). (2) Add any agent-specific business rules to BusinessRuleValidator. (3) Run the pipeline with --entity agents --table AGENT_MASTER. No code changes needed if the existing transforms cover the field types. The architecture is designed so adding entities is a configuration task.
Q22: Why is missing_in_target a critical severity but missing_in_source is only a warning?
Missing in target means data loss — a record exists in DB2 but didn't make it to FAST. This is unacceptable in insurance (missing a policy could mean an uncovered claim). Missing in source means an extra record in FAST that wasn't in this DB2 extract — it might have been loaded by a previous migration batch or created directly in FAST. Extra data is suspicious but not dangerous; missing data is catastrophic.
Q23: What's the difference between dry_run and the mock implementations?
dry_run=True runs the real pipeline but skips the LOAD stage (doesn't write to FAST). It validates extraction, transformation, and business rules against real data. Mock implementations (mock DB2, mock S3) replace external dependencies for local development and testing. You'd use dry_run in staging/pre-prod with real AWS, and mocks in local development with no AWS access.
Q24: How does the Alembic migration handle the initial schema?
The initial Alembic migration (001_initial_schema) creates all 6 tables with proper column types, foreign keys, and indexes. It uses PostgreSQL-specific features: UUID primary keys (native UUID type), JSONB columns for flexible metadata, and timezone-aware DateTime columns. The downgrade function drops tables in reverse dependency order to avoid foreign key violations.
Q25: If you could redesign one component, what would you change?
I'd make the reconciler async-capable. Currently, it loads both source and target DataFrames into memory for comparison, which limits us to datasets that fit in RAM. For billion-record tables, I'd switch to streaming comparison: read source and target in sorted key order from Parquet files on S3, compare chunk-by-chunk, and stream results to PostgreSQL. This would reduce memory usage from O(n) to O(batch_size) while maintaining the same comparison logic.
Q26: Walk me through what happens when a user asks the chatbot "Where did the premium come from?"
  1. Intent classification: "come from" triggers lineage_backward
  2. Entity extraction: "premium" matches known column "premium_amount", infers target table "fast_policies"
  3. Context retrieval: Calls lineage_engine.query_backward("fast_policies", "premium_amount") — returns entries showing it came from db2_policies.PREM_AMT via packed_decimal_to_float
  4. RAG augmentation: ChromaDB vector search finds similar lineage entries for additional context
  5. GPT-4 generation: System prompt + retrieved context + user question → natural language answer citing specific tables, columns, and transformations
  6. Response: Returns answer, intent, sources (for verification), and confidence score
